## Mutexes

A mutex is a boolean variable that indicates if a lock is available or not. To
protect a CS, a process must first acquire the lock, and then release it at the
end.

Calls to acquire and release are implemented atomically via hardware
instructions.

When waiting for the lock, the process can 2 things:

- busy waiting: it keeps spinning and checking if the lock has been released;
- context switch: the process yields control to the OS, which can schedule other
  things until the lock has been freed;

## Semaphores

A semaphore is an integer variable. It can be accessed via 2 indivisible atomic
operations:

- `wait()`: waits for a signal $> 0$ and then decrements its value by 1. Used to
  lock the resource;
- `signal()`: increments the value of the signal by $1$. Used to release the
  resource;

It can be used to give access to a finite number of resources. When the signal
goes to 0, other instances must wait for one to release its lock.

:::note

A semaphore with a max value of 1 is equivalent to a mutex.

:::

Each semaphore has an associated waiting queue where processes that call `wait`
are put. Once a process sends a signal, the processes in the queue are put into
the ready queue where they can try again to obtain the lock.

## Monitors

A monitor abstracts shared data. Only its procedures can access and modify the
shared resources and only a process between those sharing the monitor can be
active at a time.

The monitor can be seen as an API that the programmer just needs to call. There
is no need to worry about concurrency primitives. Eventually the called function
will be run in a mutually exclusive way.
