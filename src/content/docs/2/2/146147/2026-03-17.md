## Synchronization

Processes may operate concurrently on shared data, but one might be interrupted
at any point during its execution.

A process may need to read a value and operate on it with the assumption that
the value is not modified by another thread or process until it is written back
to memory.

In preemptive systems, a situation like the following may happen:

```text
process_1: register_1 = count
process_1: register_1 = register_1 + 1 // context switch
process_2: register_2 = count
process_2: register_2 = register_2 - 1 // context switch
process_1: count = register_1          // context switch
process_2: count = register_2
```

In this case, 2 processes want to update count, but in the end we won't obtain
the final value that we expect (the initial value since we sum and subtract 1)
but we get `count - 1` because the operation of process 1 is written after
process 2 has already read the value of `count`.

This is called a **race condition**.

### Critical section

The critical section of a process is the section where the process operates on
values shared with other processes.

A solution to the problem needs to satisfy 3 constraints:

- **mutual exclusion**: if a process is in its critical section, then no other
  process can access the same shared data;
- **progress**: if no process is in the critical section and one requires it,
  then the OS cannot postpone forever the request for access;
- **bounded waiting**: there must be a limit to the number of times other
  processes are allowed to enter the critical section after a process has
  required and not been granted the request to enter the critical section;

### Single core solutions

A first idea might be to disable interrupts and prevent the CPU from switching
to another process when inside the critical section. This is a bad idea because
we are giving the process the possibility to disable preemption for an
arbitrarily long time.

Next we might want to explore a software turn based solution:

```c
// process 1
while (true) {
    while (turn == i); // wait while is the turn of the other process
    // critical section
    turn = i; // assign the turn to the other process
    // remainder section
}

// process 2
while (true) {
    while (turn == j);
    // critical section
    turn = j;
    // remainder section
}
```

There is a problem, one process might never need the critical section, and so it
might never give the turn back. So if process `j` is waiting but process `i`
never uses its turn, then `j` will be waiting forever.

Peterson's solution adds a `boolean flag[2]` variable that is shared with the
other process. `turn` indicates whose turn it is to enter the critical section,
`flag` indicates is a process is ready to enter the critical section.

```c
while (true) {
    flag[i] = true;
    turn = j;
    while (flag[j] && turn == j);
    // critical section
    flag[i] = false
    // remainder section
}
```

This algorithm solves all 3 requirements of the critical section:

- mutual exclusion is preserved;
- progress is based on turn so a process is guaranteed to be able to enter the
  critical section when its turn comes back;
- bounded waiting is met because after j processes, it must allow i next,
  preventing starvation;

:::warn Peterson's solution doesn't work on modern systems:

- we may want to run more than 2 processes;
- modern compilers may reorder operations that have no dependencies to improve
  the speed of execution;

Ideally we want a single instruction for entering and exiting the critical
section.

:::

### Hardware support

There are 3 means that the hardware provides to solve the CS problem:

- memory barriers;
- hardware instructions (compare_and_swap and test_and_set);
- atomic variables;

#### Memory barriers

Memory models are the guarantees that a computer architecture makes to
application programs. A strongly ordered model ensures that one memory
modification on one process is immediately visible to others. A weakly ordered
one doesn't guarantee immediate visibility.

A memory barrier is an instruction that forces any change in memory to be
propagated (made visible) to all other processors.

When a memory barrier is used, the system ensures that all loads and stores are
completed before running any subsequent load and store.

Memory barriers are powerful but a bit slow, since their effect is global.

#### Atomic instructions

Atomic instructions allows to implement a check and set of a variable in memory,
guaranteeing that nothing else happens in between.

```c
// This is all executed atomically on the hardware. If target was already true
// then nothing changes.
boolean test_and_set(boolean *target) {
    boolean rv = *target;
    *target = true;
    return rv;
}
```

The CS becomes:

```c
while (true) {
    while (test_and_set(&lock));
    // critical section
    lock = false;
    // remainder section
}
```

The only problem here is bounded waiting, because if a process is able to
execute `test_and_set` faster than others then they will get total control of
the CS.

```c
int compare_and_swap(int *value, int expected, int new_value) {
    int temp = *value;

    // Sets new_value only if the original one is the one that we expect.
    if (*value == expected) {
        *value = new_value;
    }

    return temp;
}
```

Now the CS becomes:

```c
while (true) {
    while (compare_and_swap(&lock, 0, 1) != 0);
    // critical section
    lock = 0;
    // remainder section
}
```

This solution still has the same problem as before. To solve it we need to add
some more checks:

```c
// `lock` is initialized to 0.

while (true) {
    waiting[i] = true;
    key = 1;

    // Try to acquire the lock on the CS.
    // Process `i` can enter the CS if either `waiting[i] == false` or if
    // `key == 0`.
    while (waiting[i] && key == 1) {
        key = compare_and_swap(&lock, 0, 1);
    }

    waiting[i] = false;

    // critical section

    j = (i + 1) % n;
    // Check all other processes and see if anyone is waiting.
    while ((j != i) && !waiting[j]) {
        j = (j + 1) % n;
    }

    // If no one else is waiting then we release lock.
    if (j == 1) {
        lock = 0;
    } else {
        // Allow another process to enter it critical section by bypassing its
        // condition on waiting of the first while loop.
        // The queue stays locked and we manually pass control to the other
        // thread.
        waiting[j] = false;
    }

    // remainder section
}
```

#### Atomic variables

Atomic variables are provided from the standard library of a language and they
use atomic instructions to provide atomic operations on basic data types.
